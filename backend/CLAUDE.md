# CLAUDE.md — Backend

This file provides guidance to Claude Code (claude.ai/code) when working in `backend/`. See the root [CLAUDE.md](../CLAUDE.md) for the overall architecture.

## What this is

The GraphQL API: a Hasura GraphQL Engine (v2.x) over PostgreSQL ≥16. **There is no hand-written application code here.** The API surface is defined by two things:

- **SQL migrations** ([migrations/Postgres/](migrations/Postgres/)) — the database schema, functions, triggers, and views.
- **Hasura metadata** ([metadata/](metadata/)) — table tracking, relationships, permissions, actions, REST endpoints, cron triggers, etc.

Imperative logic that can't live in Postgres/Hasura (auth, image handling) lives in the [cloud-function](../cloud-function/), which Hasura calls as an **action handler** and **auth webhook** (see [metadata/actions.yaml](metadata/actions.yaml)).

## Setup & common commands

Requires Docker (Compose V2) and the Hasura CLI. Copy env with `cp .env.example .env`; set `HASURA_GRAPHQL_ADMIN_SECRET` and `CLOUD_FUNCTION_SECRET` to match the frontend and cloud-function `.env` files.

```bash
docker compose up -d
hasura metadata apply
hasura migrate apply
hasura metadata apply        # applied twice on purpose — ordering deps
```

Console: http://localhost/api/hasura/console (admin secret in `.env`). Auth-dependent operations also require the cloud-function running; pure admin-secret queries do not.

If the cloud-function is started **after** the backend, re-run `docker compose up -d` in `backend/` so Nginx picks up the now-reachable auth webhook.

## Changing the schema

Use `hasura migrate create <name> --database-name Postgres` to scaffold a migration under [migrations/Postgres/](migrations/Postgres/), fill in `up.sql`, then `hasura migrate apply`. Migrations are **forward-only** — delete the generated `down.sql`.

Schema changes usually need matching updates to the Hasura metadata YAML in [metadata/](metadata/) (table tracking in `databases/Postgres/tables/`, plus relationships, permissions, actions as needed), applied with `hasura metadata apply`.

**After any schema change, regenerate the frontend's GraphQL types** (`bun --bun run graphql:output` in `frontend/`).

## Domain & key tables

Entities form a hierarchy: `crossings` → `lots` → `cultivars` → `plant_groups` → `plants` (plus `plant_rows`, `orchards`, `rootstocks`, `graftings`, `mother_plants`, `pollen`). Typed `attributes` are grouped into `attribution_forms` and recorded as `attributions` against entities.

- **`cached_attributions`** denormalizes all attribution data into one table for fast Analyze queries. It was a materialized view, changed to a table for update performance (see [docs/database.md](docs/database.md) and PR #245). Full rebuild: run the `refresh_cached_attributions` mutation.
- **`logged_actions`** provides audit logging.

Read [docs/database.md](docs/database.md) before changing attribution/analyze behavior. The DB UML is regenerated by the [`database-uml.yml`](../.github/workflows/database-uml.yml) GitHub Action on push — no need to run `docs/generate-uml.sh` locally.

## Tests

GraphQL API tests run with bun's (jest-compatible) test runner against a **separate test database** that the script creates/clears.

```bash
./tests/run.sh           # full run (sets up the test env)
./tests/run.sh --quick   # reuse existing test DB (faster; not safe after schema changes)
./tests/run.sh --help    # extra args pass through to `bun test`
```

Test files live in [tests/tests/](tests/tests/), one per table/concern (e.g. `cultivars.test.js`, `cached_attributions.test.js`, `auth.test.ts`). When you add a table or change permissions, add/adjust the matching test there.
